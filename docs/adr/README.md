# 아키텍처 결정 기록 (ADR)

이 디렉토리는 이 레포의 **구조적 의사결정**을 되돌아볼 수 있게 남긴다.
ADR(Architecture Decision Record)은 "무엇을 택했나"뿐 아니라
**어떤 대안을 두고, 무엇을 포기하고, 왜 그렇게 정했는가**를 기록한다.
코드는 결정의 결과만 보여주지만, ADR은 결정의 맥락을 보존한다.

## 무엇을 ADR로 남기는가

- 되돌리기 어렵거나 여러 컴포넌트에 파급되는 선택 (경계·계약·프로토콜·통신 토폴로지 등)
- 대안이 실재했고 트레이드오프가 있었던 선택
- 나중에 "왜 이렇게 했지?"라는 질문이 나올 법한 선택

사소하거나 국소적인 구현 선택은 코드·주석·커밋으로 충분하다. ADR로 남기지 않는다.

## 서술 규칙 — 역할·포트로 쓰고, 구현 클래스명은 넣지 않는다

ADR은 승인 후 **append-only 기록**이다. 구체 구현 클래스명(transport 구현·데코레이터·기기 빌더 등)을
넣으면 rename 시 소급 수정(기록 위조) 또는 방치(부패) 중 하나가 강제된다.

- 서술은 **역할**("시리얼/소켓 transport", "재연결 데코레이터", "버스 중재 데코레이터",
  "상태 동기화 워커")과 **포트/계약**(`BaseTransport` 인터페이스, HA MQTT Discovery·토픽 계약,
  RS485 프레임 규약 등)으로 한다. 포트·계약은 결정의 대상 그 자체이므로 허용된다.
- 구체 클래스명은 [sequences.md](../sequences.md)와 코드가 소유한다 —
  살아있는 문서라 rename 시 함께 갱신된다. (전체 계층 규칙: CONTRIBUTING의 문서 계층 절)
- 대체된 ADR의 본문은 고치지 않는다 — 당시 이름 그대로 두는 것이 기록이다.

## 작성 방법

1. [`template.md`](template.md)를 복사한다.
2. `NNNN-<kebab-case-제목>.md`로 저장한다 (`NNNN`은 다음 순번, 4자리 zero-pad).
3. 아래 인덱스 표에 한 줄 추가한다.
4. 상태는 `제안됨 → 승인됨` 순으로 갱신하고, 뒤집힌 결정은 폐기하지 말고
   **`대체됨(→ ADR-NNNN)`**으로 표시해 이력을 남긴다.
